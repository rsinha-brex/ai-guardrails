Design intent. Cards read in a glance because everything operational is on the surface — rule type, active state, test coverage, and applicability. The card hierarchy puts the human-readable description front-and-center; the technical details (priority, applies_to, judge usage) live in the footer where they're available when needed but don't dominate.
02 · Create a rule

Two paths. One opinion.

A template library on the left for the canonical rule types (business hours, service area, allowed services, eligibility, lead time, communication style). A natural-language describe-your-rule textarea on the right for everything else. The compile step turns prose into structured rules with confidence scores.

Describe a custom rule

Be specific about which services and what action to block.

02
Two paths intentionally. The templates cover the canonical rules — fast for owners who know what they want. The describe-your-rule textarea handles everything else through a compile step that turns prose into structured rules. The compiled output renders as human-readable prose first, structured fields second, with an expandable "why this type" explanation. Owners always see what their words turned into before saving.
03 · Test the rule

Test-driven rule development.

Owners write scenarios — what a customer might say and what the agent should do — and run them against the rule. Failing tests expose the conversation and offer AI-assisted refinement: the system suggests a rule update with a diff the owner approves.

03
Test-driven rule development. Each rule has its own test suite. Owners write scenarios in natural language ("what a customer would say" + "what the agent should do"). Failing tests expose the conversation transcript and offer AI-assisted refinement — a diff the owner reviews and applies. This makes the rules editor a real development environment, not just a configuration form.

The agent's reasoning, on tape. The drill-in view is forensics-grade — every tool call, every rule check, every state update, every message in chronological order. Owners can verify the agent did what it should have done, or trace exactly what went wrong if it didn't. "Show agent context" reveals the full system prompt the agent had at conversation start, including the injected rule catalog — radical transparency about how the system actually works.

Designed to be honest about how it works.

Every screen surfaces the agent's reasoning rather than hiding it behind a chat box. Rules are first-class objects with test suites. The activity feed is forensic, not aspirational.
